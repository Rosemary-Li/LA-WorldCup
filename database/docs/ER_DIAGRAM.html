Schema reverse-engineered from every SELECT and JOIN in backend/queries.py. Eleven tables, seven enforced-by-query relationships. Two fact tables (fact_hotel, fact_restaurant) stand alone — they're related to matches at runtime via free-text area string matching, not by database FK.

Tables by role

  • DIM dim_team — countries, federations, group stage
  • DIM dim_player — squad lists, star flags
  • DIM dim_event_category — category labels for events
  • FACT fact_match — schedule + teams + venue
  • FACT fact_ticket — per-match ticket inventory
  • FACT fact_ranking — FIFA ranking snapshot
  • FACT fact_event — fan events / shows / attractions
  • STAND-ALONE fact_hotel — Explore LA + Journey lookups
  • STAND-ALONE fact_restaurant — Explore LA + Journey lookups
  • DETAIL event_experience_detail — 14 columns of itinerary metadata
  • DETAIL event_sports_detail — sport type / competition info

Notes on enforcement

  • Country is the natural key linking dim_team, dim_player, fact_match, and fact_ranking — no DB-enforced FK. _with_latest_team_data() in queries.py splices in renames after import.
  • event_id is both PK and FK on the two detail tables (one row per event at most).
  • event_category_id is stored as text in fact_event — note the ::int cast in get_events_by_categories().

Not in this diagram

The main README mentions three additional tables that exist in the schema but aren't queried by the current backend, so they're not drawn here:

  • dim_place — stadiums, airports, transport places
  • dim_mode — transport mode metadata
  • fact_route — airport-to-venue and local routes
